# Spec: `/clear` command for guest-mode context reset

Date: 2026-06-01

## What & why

A user mentions `@<bot> /clear` in a chat to wipe their own conversation
context with the bot in that chat. The next message the bot answers starts
"from a clean slate" — no prior history is fed to Groq.

Today history is keyed by `(chat_id, user_id)` in SQLite (`bot/memory/store.py`)
and fed into the prompt via `store.get_history`. There is no way for a user to
reset it short of waiting for the TTL prune.

## Scope

- **Clears only the caller's history in the current chat** — the
  `(chat_id, user_id)` scope. Other participants in the same chat are not
  affected. This matches how the store is already keyed.
- Out of scope: chat-wide clears, admin-only clears, per-message undo,
  configurable confirmation text. (YAGNI.)

## Behaviour

1. Bot receives a `guest_message`; `strip_bot_mention` removes the `@<bot>`
   mention (this also collapses the `/clear@<bot>` form to `/clear`).
2. If the stripped text **is exactly** the clear command, the bot:
   - deletes the caller's history for `(chat_id, user_id)`,
   - answers **once** via `answer_guest_query` with a fixed confirmation,
   - returns early — Groq is NOT called and no history is appended.
3. Otherwise the existing flow runs unchanged.

**Command match:** exact only. After stripping the mention, the text must
equal `/clear` ignoring case and surrounding whitespace. Any extra text
(e.g. `/clear please`) is treated as a normal question, not a command.

**Confirmation reply:** a short fixed string, e.g.
`🧠 Context cleared — starting fresh.` Not configurable, not generated by Groq.

## Components

1. **`MemoryStore.clear(chat_id, user_id)`** (`bot/memory/store.py`) — new
   method: `DELETE FROM messages WHERE chat_id=? AND user_id=?` + commit.
   Symmetric with `append` / `prune`.

2. **`is_clear_command(text) -> bool`** (`bot/telegram/guest.py`) — pure
   function: `text.strip().casefold() == "/clear"`.

3. **`CLEAR_REPLY`** constant in `guest.py`, next to `FALLBACK_TEXT`.

4. **Branch in `handle_guest_message`** (`bot/telegram/guest.py`), right after
   `strip_bot_mention`:

   ```python
   user_text = strip_bot_mention(gm.text, config.bot_username)
   if is_clear_command(user_text):
       await store.clear(gm.chat_id, gm.user_id)
       await api.answer_guest_query(gm.query_id, CLEAR_REPLY)
       return
   # ... existing flow
   ```

## Error handling

No special handling. If `store.clear` raises, the exception propagates to
`bot/telegram/webhook.py`, which already logs and returns 200 to avoid
Telegram retry storms.

## Tests (TDD)

- `store.clear` deletes only the `(chat_id, user_id)` scope and leaves another
  user's / another chat's history intact.
- `is_clear_command`: matches exact `/clear`; case-insensitive; matches the
  `/clear@<bot>` form *after* `strip_bot_mention`; rejects `/clear please` and
  empty / unrelated text.
- `handle_guest_message` on `/clear`: calls `store.clear` once, calls
  `answer_guest_query` once with `CLEAR_REPLY`, and does NOT call
  `ai.stream_completion` nor `store.append`.
